# Google Login With User Bearer Tokens

## Objective

Document what still needs to be added so Google login can behave like credential login in the current frontend architecture:

- keep `sid` as the local web session
- also obtain the real user `access_token`
- also obtain the real user `refresh_token`
- store those tokens server-side in `httpOnly` cookies
- let Next route handlers use the user bearer instead of the shared technical token

This document is about the missing implementation plan, not the current social login diagnosis.

## Current State

Today the Google popup flow is enough to complete federated auth and, when backend works correctly, create local session cookie `sid`.

That solves app access and middleware, but it does **not** currently guarantee that frontend proxies can call protected BFF endpoints with the real user identity.

Because of that, the frontend still falls back to a technical token configured through:

- `FSA_AUTH_TOKEN_URL`
- `FSA_AUTH_TOKEN_USERNAME`
- `FSA_AUTH_TOKEN_PASSWORD`

That fallback is useful to unblock development, but it is not the desired final behavior.

## Target Behavior

After a successful Google login, the system should end with both:

1. local app session
   - `sid`

2. user bearer session
   - `access_token`
   - `refresh_token`
   - token expiration metadata

Then Next route handlers should:

- read the user bearer from `httpOnly` cookies
- use it for `/portal/me/perfil`
- use it for `/portal/me/expedientes`
- use it for `/logistica/*`
- refresh it automatically when needed
- keep the technical token only as temporary fallback while migration is incomplete

## Why Google Needs A Different Hook Point

With username/password login, the frontend already has direct credentials at submit time, so the server can exchange them for bearer tokens during `/api/auth/login`.

With Google login, the frontend never sees user credentials.

Because of that, the bearer tokens must be captured at a different point:

- when the backend processes the OAuth callback successfully

That is the critical hook for Google.

## What Backend Must Provide

For Google login to support real user bearer tokens, backend must do one of these two things.

### Option A: Return tokens during callback success

After backend validates OAuth callback and creates local session, it should also make available:

- `access_token`
- `refresh_token`
- `expires_in`
- `refresh_expires_in`

This can happen through:

- response body
- response headers
- or direct `Set-Cookie` generated by backend itself

Best case is backend already sets secure `httpOnly` cookies directly.

### Option B: Expose an exchange endpoint after session creation

If backend does not want to return tokens from the callback response, then it should expose a server-side endpoint such as:

- `POST /auth/session/token`
- or similar

The purpose of that endpoint would be:

- read the just-created authenticated session
- return bearer tokens for that same user

That exchange must happen only on trusted server-side calls.

## Recommended Approach

Preferred approach:

1. Google callback finishes successfully.
2. Backend creates `sid`.
3. Backend also returns user `access_token` and `refresh_token`, or sets them directly in secure cookies.
4. Next callback proxy preserves those cookies.
5. Frontend main window verifies session exactly as today.
6. Protected frontend proxies start using the user bearer automatically.

This keeps the existing popup UX and does not require replacing `sid`.

## Frontend Changes Needed

The following behavior should exist in frontend once backend supports it.

### 1. Preserve token cookies in Google callback proxy

Current Google callback proxy already forwards upstream headers and `Set-Cookie`.

Relevant file:

- `src/app/api/auth/providers/google/callback/route.ts`

If backend starts setting user token cookies directly, frontend may need no extra cookie parsing logic there, only to ensure those cookies are preserved exactly like `sid`.

If backend returns tokens in body instead of cookies, the callback proxy will need to:

- parse upstream response
- create `httpOnly` cookies in Next response
- preserve popup success HTML / redirect behavior

### 2. Define standard cookie names for user tokens

Frontend server-side token handling should keep using `httpOnly` cookies such as:

- `fsa_access_token`
- `fsa_access_token_expires_at`
- `fsa_refresh_token`
- `fsa_refresh_token_expires_at`

Those names are already aligned with the current credential-login implementation.

### 3. Keep `sid` untouched

No need to replace session cookie.

`sid` should still be used for:

- app authentication state
- middleware access checks
- redirect logic
- popup success verification through `/api/auth/session`

### 4. Use user bearer first inside Next proxies

Relevant files:

- `src/app/api/_lib/proxy.ts`
- `src/app/api/portal/me/perfil/route.ts`
- `src/app/api/portal/me/expedientes/route.ts`
- `src/app/api/logistica/[cicloId]/parent-orders/route.ts`

Expected precedence:

1. incoming `Authorization` header
2. user bearer from cookies
3. refreshed user bearer from `refresh_token`
4. technical token fallback

### 5. Clear user token cookies on logout

Relevant file:

- `src/app/api/auth/logout/route.ts`

Logout must keep clearing:

- `sid`
- `trusted_device_token`
- all user token cookies

## Google Popup Flow After This Change

The popup flow should remain functionally the same for the UI.

### User-facing sequence

1. User clicks `Continuar con Google`.
2. Popup opens.
3. Google + Keycloak authenticate user.
4. Backend callback creates `sid` and emits user bearer tokens.
5. Next callback proxy preserves those cookies.
6. Popup sends `SOCIAL_AUTH_SUCCESS`.
7. Main window verifies session with `/api/auth/session`.
8. User lands in `/home`.
9. Subsequent frontend proxies call BFF with the real user bearer.

## Security Notes

The frontend should **not**:

- persist Google-derived bearer tokens in `localStorage`
- expose `refresh_token` to browser JavaScript
- store user password for later token exchange

The correct place for these tokens is secure `httpOnly` cookies handled on the server side.

## Backend Contract Checklist

Backend support for Google should confirm all of the following:

- callback success creates `sid`
- callback success also exposes user `access_token`
- callback success also exposes user `refresh_token`
- expiration metadata is available
- token audience and roles are the real user ones
- cookies are compatible with popup flow and frontend domain
- refresh flow works server-side without requiring frontend credentials

## Frontend Validation Checklist

When implemented, validate this exact sequence:

1. Login with Google.
2. Confirm `sid` is present.
3. Confirm user token cookies are present.
4. Call `/api/portal/me/perfil` and verify it uses user roles.
5. Call `/api/portal/me/expedientes` and confirm `cicloActual` is resolved with user permissions.
6. Open `Segui tu pedido` and confirm logistics uses the user bearer.
7. Force access token expiry and verify refresh works.
8. Logout and confirm `sid` and token cookies are removed.

## Interim Status

At the time of writing:

- credential login already supports storing user bearer tokens in server cookies
- protected proxies already know how to prioritize the user bearer before the technical token
- Google login still depends on backend exposing user tokens after callback success

So the missing piece is specifically the Google callback token handoff.

## Short Summary

To make Google login work with the real user bearer, do not replace `sid`. Keep `sid` for the web session, and extend the Google callback so backend also provides `access_token` and `refresh_token` for that same user. Once those tokens are stored in secure `httpOnly` cookies, the existing Next proxies can call the BFF with the real user identity instead of the shared technical token.
